TRoOGiverse — Rules of Origin Analytics Tool

An interactive Shiny application that makes thousands of complex trade rules searchable and easy to navigate, built for users with no technical background as well as expert analysts.

Built with R Shiny, tidyverse, dplyr
Type Interactive Reference Tool
Data scope Thousands of product and country rules
Audience Technical and non-technical users

Overview

Trade rules are complicated. Each product category can have different qualifying conditions depending on where it is being traded, what agreement is in place, and how it was made. Looking these up manually is slow and often leads to errors, especially for people who are not specialists in this area.

TRoOGiverse solves this by putting all of those rules into a searchable, interactive tool. Users can type in a product code or country and instantly see the relevant conditions, without needing to read through legal documents or ask an analyst for help every time.

What it does

  • Fast search across thousands of rules

    Users can search by product code, country, or rule type and get the relevant information instantly. No scrolling through long documents or relying on someone else to look it up.

  • Filter by country, product, and rule type

    Users can narrow results using multiple filters at once. Everything updates instantly as filters are applied, with no page reloads or waiting.

  • Structured data behind a simple interface

    The underlying rules data is complex, with many conditions, exceptions, and variations across products and countries. All of that was structured and modelled in R before being surfaced through a clean, easy to use interface.

  • Built for everyone, not just analysts

    The design was deliberately kept simple so that non-analysts and colleagues across the organisation could use it independently without needing training or support from a data analyst.

  • Stays fast as the dataset grows

    With thousands of product and country combinations, keeping the app responsive required careful design choices around how data is stored and retrieved. The app was built to stay fast even as the reference data expands.

Challenges

The hardest part of this project was organising the underlying data. Trade rules are not uniform. Different products have different types of qualifying conditions, some have multiple alternatives, and some have exceptions that only apply in certain situations. Getting all of that into a consistent structure that could be queried cleanly took significant data modelling work before any of the app interface could be built.

This was also a collaborative project. Each analyst on the team built their own individual scripts covering different parts of the rule set. The challenge was then bringing all of those separate scripts together into a single central script that could pull from each one cleanly, without creating multiple versions of the same logic or risking one change breaking everything else. Building that structure required careful coordination and design so the whole team could keep working independently while the combined output stayed reliable.

Keeping the app fast as the number of combinations grew was also a challenge. This was handled by pre-filtering at the data level and making sure the app only recalculates what it needs to when a user changes a filter.
